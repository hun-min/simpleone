# 작업 규칙

## 필수 규칙

**수정 전**:
1. 관련 파일 먼저 읽기
2. **중복 코드 확인 필수** - 같은 기능이 여러 파일에 있는지 검색
   - 예: 팝업이 SmallPopups.js와 App.js 둘 다 있는지
   - 예: 프로토콜 화면이 여러 곳에 있는지
3. "이렇게 수정하겠습니다. 할까요?" 물어보기
4. 사용자 승인 후 작업
5. 코드 수정 후 배포 전 반드시 변경사항 요약 보고
   - 예: "maxWidth: 500px → 600px", "'뇌 충격으로 깨우기' → '뇌 깨우기'"

**배포 전**:
1. 변경사항 요약 보고 (무엇을 어떻게 바꿨는지)
2. "배포할까요?" 물어보기
3. 사용자 승인 후 배포
4. "배포 ㄱ" 하면 바로 배포

**배포 확인**:
- 사용자는 배포할 때마다 새 Vercel URL로 확인함 (캐시 문제 없음)
- 예: https://simpleone-[랜덤문자열]-hun-mins-projects.vercel.app/
- 캐시 문제라고 말하지 말 것

**절대 금지**:
- 사용자 확인 없이 수정/배포
- 파일 읽지 않고 작업
- **중복 코드 확인 없이 작업** - 한 곳만 수정해서 안 되는 경우 방지
- 핵심 기능 임의 제거
- 같은 질문 2번 이상 반복
- 거짓말 (모르면 "모릅니다")
- 안 되면 같은 방법 반복 - 다른 원인 찾아야 함

**CSS 스타일 수정 시 필수**:
1. 인라인 스타일 수정 전 반드시 App.css의 기본 스타일 확인
2. CSS 우선순위: 인라인 스타일 > CSS 파일
3. maxWidth만 설정하면 CSS의 width가 우선 적용될 수 있음
4. 예: .popup { width: 280px } 있으면 maxWidth: 600px 해도 280px로 보임
5. 해결: 인라인에 width 명시 (예: width: '600px', maxWidth: '90vw')

**커밋 규칙**:
- 모든 커밋 메시지는 한글로 작성
- 예: "공간 기능 추가", "버그 수정"

**존댓말 필수**:
- 사용자가 반말을 극도로 싫어함
- "~함", "~됨" 같은 반말 절대 금지
